# Pattern: Security Through Filtering

## Problem

When agents fetch web content, raw HTML enters the context window. Adversarial content in that HTML can attempt prompt injection. The human using an agent-facing tool shouldn't have to worry about attacks on the agent.

## Solution

The tool is a filter between the data source and the agent. The agent never sees raw external content — only structured, bounded responses generated by the tool from curated data.

## Threat Model

| Attack vector | How it works | How filtering prevents it |
|---|---|---|
| Prompt injection via content | Adversarial instructions embedded in web page text | Agent never sees page text — only tool-generated summaries |
| Injection via user-generated content | Malicious input in comments, descriptions, etc. | Data is pre-curated; user-generated content doesn't enter the pipeline |
| Data poisoning | Attacker modifies the source data | Source data is in a repo the steward controls (PRs, access control) |
| Response manipulation | Adversarial formatting in API responses | Tool generates all formatting; data values are inserted into controlled templates |

## Implementation

1. **Data lives in a controlled repository.** The steward (human) owns the repo. Changes go through git. No external writes.
2. **The tool reads the data file, not the network.** No HTTP requests, no API calls, no dynamic content. The data is a local file.
3. **Responses are generated by the tool, not assembled from data.** The tool templates the output; data values fill slots. No data value can alter the response structure.
4. **No raw content passthrough.** Summaries and skeletons are pre-written by the steward or a trusted process. They're not fetched at query time.

## Tradeoff

This pattern means the data is only as fresh as the last update to the data file. For a knowledge graph that changes slowly, this is fine. For real-time data, a different pattern is needed — but the filtering principle (agent never sees raw external content) still applies.
